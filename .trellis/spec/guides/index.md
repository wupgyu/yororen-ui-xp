# 思考指南

> **作用**：扩展你的思考范围，帮助你提前发现可能遗漏的问题。

---

## 为什么需要思考指南？

**大多数 bug 和技术债并不是因为能力不够，而是因为“没想到这一层”**：

- 没想到层与层之间会怎样交互 → 跨层 bug
- 没想到代码模式正在重复 → 到处都是重复代码
- 没想到边界情况 → 运行时错误
- 没想到未来维护者 → 代码难以理解

这些指南的目标，是帮助你在编码前**问对问题**。

## 快速参考：什么时候该主动思考

### 什么时候该检查跨层问题

- [ ] 功能会同时触及 3 层及以上（API、Service、Component、Database）
- [ ] 数据格式在不同层之间发生变化
- [ ] 多个消费方需要同一份数据
- [ ] 你不确定某段逻辑应该放在哪一层
- [ ] 你正在新增 event kind、JSONL record、RPC payload 或配置字段
- [ ] UI / 命令层代码开始直接强转原始 payload 字段

→ 阅读 [跨层思考指南](./cross-layer-thinking-guide.md)

### 什么时候该检查代码复用

- [ ] 你写的代码和现有代码很像
- [ ] 你看到相同模式重复了 3 次以上
- [ ] 你正在给多个位置新增同一个字段
- [ ] **你正在修改任何常量或配置**
- [ ] **你准备新建 utility/helper 函数** ← 先搜索现有实现！
- [ ] 两个文件都在用本地强转读取同一份未类型化 payload 字段
- [ ] 多个分支都在根据 \`kind\` / \`action\` 更新同一份派生状态

→ 阅读 [代码复用思考指南](./code-reuse-thinking-guide.md)

### 什么时候该复核 AI 交叉审查结果

- [ ] 审查器说“用户输入可能是恶意的” → 先确认真实数据源（内部清单？用户配置？外部 API？）
- [ ] 审查器说“缺少校验” → 先判断数据是不是来自可信内部来源
- [ ] 审查器说“行为变化了” → 先看代码注释，这是否本来就是刻意设计
- [ ] 审查器在测试里指出“bug” → 心里先删除被测功能，再看测试是否仍然能过；如果能，那就是自证型测试

**AI 审查常见误报模式**：
1. **信任边界混淆**：把内部数据（如随包分发的 JSON manifest）误当成外部不可信输入
2. **忽略设计注释**：把代码注释里明确说明的有意行为当成 bug
3. **变量理解错误**：没有一路追踪变量的真实定义（例如 Map 实际按 path 而不是 name 建索引）

**复核规则**：每个 CRITICAL/WARNING 结论都必须先回到真实代码核对后再决定优先级。AI 审查保守估计约有 35% 误报率。

## 修改前规则（关键）

> **改任何值之前，一定先搜索！**

```bash
# 先搜索你准备修改的值
grep -r "value_to_change" .
```

这个习惯能避免大多数“漏改另一处”的 bug。
